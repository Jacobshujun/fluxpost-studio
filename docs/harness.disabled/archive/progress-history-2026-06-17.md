## Risks

- The repository is not currently a Git repository from this working path, so changes cannot be inspected with `git status`.
- Runtime data is SQLite-based under `data/fluxpost.db` unless `DATABASE_URL` is configured; Node.js 24+ is required for the SQLite path because it uses `node:sqlite`.
- PostgreSQL runtime support is implemented but a live migration has not been executed in this workspace. The next live step requires a reachable PostgreSQL service and `DATABASE_URL`.
- The current PostgreSQL table design preserves existing JSONB-backed store behavior; it is not yet the final high-volume queue model. Queue workers with `FOR UPDATE SKIP LOCKED` remain a follow-up.
- External services TikHub, OpenAI-compatible relay, and Feishu CLI are intentionally excluded from default baseline checks.
- Real GPT text/vision tagging depends on the configured OpenAI-compatible relay. If the relay rejects image inputs, visual tagging records a failed status while content text tagging can still succeed.
- Remote platform image URLs can expire or reject browser requests. The media proxy improves browser preview for reachable remote images, but truly expired signed URLs still require recrawling/backfilling to recover.
- The new media backfill retries existing remote URLs but cannot recover already-expired or protected URLs; those still require recrawling or a platform-specific extraction improvement.
- Video backfill requires a direct downloadable video URL and `ffmpeg` on PATH for frames.
- Video frame extraction depends on `ffmpeg` being installed on PATH.
- `npm run build` passes, but Turbopack reports known tracing warnings involving `src/lib/media-cache.ts` and `next.config.ts`.
- Next production static serving should not be relied on for media files created after build time. Local crawled/generated media is now served by the app route and rewrites, so `/media/crawl/...` and `/generated/...` remain stable browser paths after runtime writes.
- If `http://127.0.0.1:3001` appears to serve old frontend/API code after a build, confirm `scripts/local/restart.ps1` stopped the real listener PID. Earlier Windows runs printed the npm parent PID while the old Node listener stayed alive.
- Material library assets are local path indexes; browser preview only works for HTTP URLs or app-served public paths until a server-side preview endpoint exists.
- Crawl, single generate, and batch generate APIs currently respond only when backend work is complete. The new progress meters therefore show explicit frontend phase progress during the request and final real counts after response. True per-item live progress needs an async backend job model with polling or server push.
- `node:sqlite` emits an expected experimental warning in Node 24; the warning did not fail build or baseline verification.
- The local environment had `HTTPS_PROXY=http://127.0.0.1:9`, which breaks Feishu network checks. `lark-cli doctor` and Base checks pass when `LARK_CLI_NO_PROXY=1` and proxy env values are cleared. The Feishu wrapper clears that specific broken proxy for CLI subprocesses.
- On Windows, Node `spawn`/`execFile` does not reliably execute the npm `lark-cli` shim. Use the wrapper's resolver rather than calling `execFile("lark-cli", ...)` directly.
- On Windows, Node treats root-relative web paths like `/media/...` as absolute disk paths. Feishu attachment resolution must map these app URLs to `public/` before calling `path.isAbsolute`.
- `lark-cli base +record-batch-create --json @file` requires a relative path under the current working directory and strict UTF-8 without BOM. The wrapper now passes relative JSON payload paths.
- Real Feishu write permission is still not verified by an actual record creation; only read access and dry-run command shape were verified.
- Feishu IM notification needs a configured recipient and bot message permission. Without `FEISHU_NOTIFY_CHAT_ID` or `FEISHU_NOTIFY_USER_ID`, Base publishing can still succeed and the UI will report that notification was skipped. If both are configured, chat notification is preferred.
- Bulk delete for content items and generated posts is destructive. The UI requires browser confirmation, but there is no undo/restore feature yet.
