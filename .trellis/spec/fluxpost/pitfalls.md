# Pitfalls

Last updated: 2026-07-20

## Known Project Pitfalls

- `.env.local` contains secrets and must not be read into Trellis docs or final answers.
- Do not store production UI-managed advanced config only in the app container writable layer. Docker production uses `FLUXPOST_CONFIG_FILE=/app/config/.env.local` backed by the `fluxpost-config` named volume; removing that volume loses admin-managed secrets and overrides. `shared/env.production` remains the deployment-owned base layer.
- Fresh Ubuntu bootstrap intentionally keeps Caddy stopped and app port 3101 bound to `127.0.0.1` until a domain is enabled. Do not change the binding to `0.0.0.0` as a shortcut; use an SSH tunnel before DNS/HTTPS. Existing servers without `FLUXPOST_PROXY_ENABLED`, `FLUXPOST_PUBLIC_HOST`, or `FLUXPOST_APP_PORT` rely on proxy-on, `bbs.vollov1.xyz`, and `3101` compatibility defaults.
- Compose interpolation does not automatically consume a service `env_file`. Deployment commands must pass `--env-file /opt/fluxpost-studio/shared/env.production` (or the release symlink) so the configurable app port and Caddy hostname resolve consistently.
- Never use `docker compose down -v` during install, update, rollback, or cleanup. It removes PostgreSQL, advanced-config, runtime-data, generated/crawled-media, node-home, and Caddy certificate volumes.
- For local PostgreSQL debugging, use `npm run db:diagnose` and the dedicated read-only `fluxpost_reader` connection from `FLUXPOST_DIAG_DATABASE_URL`. Do not read or print `.env.local`, `DATABASE_URL`, or app write-role credentials just to inspect runtime state.
- The project path is now expected to be a Git repository initialized from the local source snapshot. Local secrets, runtime data, generated/cached media, debug JSON, `test-artifacts/`, `.next/`, and `node_modules/` still exist on disk in this workspace, so verify `.gitignore` and staged files before every push.
- Runtime data is stored in SQLite at `data/fluxpost.db` by default; when `DATABASE_URL` is configured the app uses PostgreSQL instead. Legacy JSON files under `data/` are migration/compatibility artifacts.
- `src/lib/concurrency.ts` provides application-process concurrency control only. It limits provider pressure while the current Node process is alive, but it is not a durable PostgreSQL job queue and does not resume in-flight provider tasks after a server restart.
- `simple_run_queue` persists simple-mode queued work and worker leases, but it does not yet provide subtask-level replay for a run already inside crawl/production/publish. Do not blindly restart a stale running simple run after a crash; until Feishu records are idempotently mapped per post, publish-stage replay could duplicate Base records.
- Simple-run force termination closes the local run and queue state and prevents later workflow saves from reviving that run. It does not guarantee immediate cancellation of an external TikHub/OpenAI-compatible image-provider/Feishu HTTP or CLI call that is already in flight inside the Node process; the worker stops at the next workflow boundary after the awaited call returns.
- Feishu publish jobs now enqueue through `feishu_publish_queue`, but a queue state change does not cancel a Feishu CLI subprocess already in flight. The worker records terminal state after the CLI returns.
- A running Feishu publish job with an expired worker lease is marked failed/interrupted instead of being automatically replayed. This avoids accidental duplicate Base records if a process died after record creation but before local state was fully persisted.
- The Feishu publish queue serializes writes per `owner_user_id`. New signed-in simple/manual publishes use the workspace account id as owner; historical simple runs without `ownerUserId` still fall back to `local`, so old queued or replayed work can still share the legacy lane.
- Do not rewrite generated-post ownership to the Feishu queue owner during publish enqueue. Admins can see and publish other members' posts, but those posts must keep their original `ownerUserId`/`ownerDisplayName`; changing them to the current queue owner can make local generated-post persistence fail with `Generated post not found` before Feishu work is queued.
- Default workspace access is whitelist mode for a small local team. It partitions app-layer content by `ownerUserId`: normal members see only their own content pools, generated posts, material library entries, execution logs, crawl jobs, simple runs, runtime posts, and Feishu publish jobs; admins see all records.
- Unowned legacy records are effectively admin-only for normal member reads because member filters require `ownerUserId === account.id`. Do not silently assign historical runtime records to a member without an explicit user request.
- `WORKSPACE_ALLOWED_USERS`, `WORKSPACE_ADMIN_USERS`, `WORKSPACE_ACCESS_PASSWORD`, and account passwords are sensitive local access configuration. Do not copy the real whitelist, admin list, setup key, or passwords into Trellis docs or final answers.
- `LARK_TASK_API_TOKEN`, `LARK_TASK_USER_MAP`, and real `LARK_TASK_CHAT_IDS` are sensitive local task-launch configuration. Do not copy real values into Trellis docs or final answers. Treat `LARK_TASK_USER_MAP` as access control because it maps Feishu senders to workspace owners.
- App data/API lists are owner-scoped, but browser media paths under `/media/...` and `/generated/...` are still direct local file URLs if a user already knows a path. Treat media URL authorization as a remaining V1.1 hardening item, not as solved by owner filtering.
- Most mutating workspace API routes now return `401` without a valid workspace session before parsing request bodies, writing local runtime state, or calling external providers. Default verification should use read-only endpoints or the documented guarded unauthenticated smoke check, not unauthenticated POSTs as setup helpers.
- Current private workspace reads also make `npm run local:watch-simple -- -Once` return `401` without an authenticated browser/session context. Before a restart, use an authenticated UI/API check or a read-only database status query to confirm there are no intentionally running simple tasks.
- Do not create real whitelist credentials or local account-table credentials in automated verification. Verification should use static checks and read-only `/api/accounts/session` unless the user explicitly asks for a live credential test.
- Do not nest long-running work inside the same provider pool that its child HTTP calls need. For example, platform crawl fan-out should not hold a crawl-pool slot while the nested TikHub requests are waiting for crawl-pool slots.
- The SQLite implementation uses Node.js built-in `node:sqlite`, so the app requires Node.js 24+ for the current local database path.
- PostgreSQL migration copies database rows only. It does not move crawled/generated media files under `public/media/` or `public/generated/`.
- Concurrent simple-run production for the same owner/project can still create PostgreSQL contention in `markSourceRewritten(...)`, because it reads and rewrites the content-project JSON row after generated-post persistence. The simple workflow now records the generated post in the run snapshot before source-status sync, and `markSourceRewritten(...)` retries PostgreSQL `40P01`/`40001` conflicts. If source-status sync still fails after retries, the draft should remain visible and Feishu publish enqueue should continue; the remaining symptom is a source-item usage/status warning, not a hidden generated draft.
- Do not change simple-run publish preparation back to `Promise.all(approvedPosts.map(...))`. A real 小鹏P7+ link-mode run (`simple-1781160663254`) generated 10 drafts but failed before Feishu enqueue with PostgreSQL `检测到死锁` when approved-post local persistence was fanned out concurrently.
- Do not change Feishu publish queue local post persistence back to `Promise.all(...persistOnePost...)`. A real 小鹏G6 Xiaopeng BBS link-mode run (`simple-1781251944027`) generated 17 drafts but failed before durable Feishu enqueue with PostgreSQL `timeout exceeded when trying to connect` when Feishu queue publish-prep persistence fanned out local DB work.
- Do not change normal execution logging back to read/full-table rewrite. PostgreSQL logs have shown concurrent `execution_logs` full rewrites can produce duplicate primary-key errors and deadlocks; `recordExecutionLog(...)` should append one row through `appendExecutionLogToDb(...)` and trim old rows.
- `npm run db:migrate:postgres -- --dry-run` reads SQLite row counts without requiring PostgreSQL. Running the real migration requires `DATABASE_URL` in the shell and a reachable PostgreSQL service.
- `DATABASE_URL` is sensitive configuration. Do not print it, copy it into Trellis docs, or include it in final answers.
- Material library assets store local file paths as indexes. Browser preview only works for HTTP URLs or app-served public paths unless a server-side preview endpoint is added.
- `src/lib/media-cache.ts` calls the system `ffmpeg` binary. If `ffmpeg` is not on PATH, video frame extraction can fail.
- Video-frame perceptual hashes are computed locally with `ffmpeg` from extracted frame files. If hash computation fails for a frame, the shared selector falls back to the older score/time/URL policy for that frame instead of blocking media caching.
- `npm run build` currently passes but emits Turbopack tracing warnings around dynamic filesystem paths in `src/lib/media-cache.ts`, `src/lib/image-generation.ts`, `src/lib/comfyui-klein.ts`, Feishu/local-media helpers, and import traces mentioning `next.config.ts`.
- `next start` does not hot reload. After frontend or API code changes, running only `npm run build` can leave an existing `http://127.0.0.1:3001/` process serving old assets. Use `npm run local:restart` to rebuild, stop the old port-3001 process, restart the server, and run local smoke.
- Next production static serving should not be trusted for files created under `public/` after build time. Runtime crawled/generated media must be served through the local media API rewrites, currently `/media/crawl/:path*` and `/generated/:path*` to `/api/media/local/[...path]`.
- Repaired runtime media files may keep the same browser URL. Because local media responses are cacheable, an already-open browser can keep showing old broken bytes unless the frontend preview URL includes the current local media cache-bust query or the browser cache is bypassed.
- On Windows, the process printed by `Start-Process npm.cmd` can be the npm parent, not the Node process listening on port `3001`. If old frontend/API code still appears after restart, check the real listener PID with `netstat -ano -p tcp`; `scripts/local/restart.ps1` now uses this as a fallback and verifies the port is free before starting.
- `data/feishu-outbox/` may be created by publishing when Feishu CLI is not configured.
- The local Feishu CLI command is `lark-cli`; current Base writes use `lark-cli base +record-batch-create`, not the older README example `bitable import`.
- On Windows, a Next/Node server may fail with `spawn lark-cli ENOENT` even when PowerShell can run `lark-cli`. The app wrapper must resolve the npm CLI shim to `@larksuite/cli/scripts/run.js` and invoke it with Node.
- A broken proxy value such as `HTTPS_PROXY=http://127.0.0.1:9` makes `lark-cli doctor` and Feishu writes fail. Use `LARK_CLI_NO_PROXY=1` or clear proxy env values for Feishu CLI commands.
- Feishu Base attachment fields such as `动态素材` cannot be written as normal record cell values. Create the record first, then upload local files with `lark-cli base +record-upload-attachment`.
- Feishu task-number content import reads `动态素材` as a Base attachment field. The cell value may only contain attachment metadata/file tokens, so imported materials must be downloaded with `lark-cli base +record-download-attachment`; do not treat attachment cell values as browser URLs or normal text.
- The `/distribution-check` page performs live Feishu writes when the signed-in operator clicks the action. Default verification must use `.trellis/verification/distribution_check.mjs` and must not post real record numbers to the API/page, because that would update `是否分发` and `内容评分` on the target Base.
- `/distribution-check` has an operator-editable audit prompt saved in workspace settings. Treat prompt edits as judgment guidance only; do not remove the backend JSON-only, allowed-value, and uncertain-means-`不可分发` guards when changing this tool.
- Current `lark-cli base +field-list` does not support `--format json`; use `--jq "."` when the app needs parseable JSON from field-list. `+record-search` and `+record-get` do support `--format json`.
- Distribution audit treats `动态素材` only as attachment metadata for a material-count signal. Do not add attachment download/upload to this tool; it should write only the `是否分发` single-select field and `内容评分` numeric field.
- Source-link import no longer mirrors harvested content into a Feishu source-import Base. Do not reintroduce a link-import Feishu write side effect; generated-post publishing and explicit Feishu task-number import remain the Feishu paths.
- Dongchedi is a source-link/ID import platform only. Direct Dongchedi article HTML may return ByteDance/Dongchedi anti-bot challenge JavaScript instead of article data; the importer should fail clearly in that case rather than producing an empty source item. Default verification must use `.trellis/verification/dongchedi_import_check.mjs` and no-network fixtures, not a live Dongchedi URL import.
- `npm run lark:tasks` can read real Feishu/Lark messages when `LARK_TASK_CHAT_IDS` is set. Default verification must use `node scripts/lark-task-runner.mjs --self-test` and `.trellis/verification/lark_task_launcher_check.mjs`, not a live chat poll. Live `--reply` sends visible Feishu replies and should only be used when intentionally testing the configured chat.
- Large simple-mode Feishu publishes can fail during `record-upload-attachment` after Base records have already been created. Do not blindly rerun the whole publish for such a run; first confirm existing records/attachments or add an idempotent attachment-resume path, otherwise duplicate Base records are possible.
- Current Feishu publish stores per-post `feishu.recordId` and attachment status when attachments fail. Retry through generated posts that preserve this state so existing Base records can be reused and already uploaded attachments can be skipped; do not clear `post.feishu` before retrying.
- Feishu CLI errors and activity-log messages should redact bearer tokens, `--base-token` values, and `FEISHU_BITABLE_APP_TOKEN` values. If a new CLI path is added, keep token redaction in the error path.
- For Feishu attachments, app-served URLs beginning with `/media/` or `/generated/` are browser paths and must be resolved under project `public/`, not treated as Windows disk-root paths like `C:\media\...`.
- For Feishu attachments, reachable remote HTTP(S) image/video URLs must be materialized before Base record creation and cleaned afterward. `lark-cli base +record-upload-attachment` rejects absolute `--file` paths even when they exist, so temporary copies must stay under the app working directory (currently `data/feishu-outbox`) and be passed as `./...` relative paths; the OS `/tmp` directory is not a valid staging root for this CLI boundary.
- `lark-cli base +record-batch-create --json @file` rejects absolute `@` paths and UTF-8 BOM files; use a relative path under the project root and UTF-8 without BOM.
- `lark-cli` exit code `0` plus `record_id_list` is not sufficient evidence that generated-post fields were written. Production `1.0.67` created rows with blank `动态标题`/`动态正文` while returning IDs. Generated-post publishing must repair each created/reused row with `+record-upsert`, batch-read the expected fields with `+record-get`, and return `record_failed` while preserving record IDs when read-back mismatches.
- `lark-cli@1.0.67 base +record-get` returns a single-select field as a one-item array even when field metadata says `multiple=false`. Generated-post read-back verification must treat a scalar write value as equivalent only to a one-item array containing that value; empty, multi-item, or different-value arrays remain mismatches.
- A persisted Feishu record ID may no longer exist even when `+record-upsert --record-id` exits `0`. Create a replacement without `--record-id` only when `+record-get` explicitly returns that ID in `record_not_found`; persist the replacement ID and verify it again. Malformed reads, omitted rows, and field mismatches must remain `record_failed` without creating another row.
- Root `.tmp-*.json` files and `test-artifacts/` are local artifacts, not project memory.
- Crawled `downloadedImages` can be a partial local cache and may not have the same count as remote `images`; do not infer image coverage from `downloadedImages.length`.
- Xiaohongshu CDN images can display as empty in the browser when requested with the local app as Referer; image elements should keep `referrerPolicy="no-referrer"` and previews should prefer cached local media when available.
- Remote browser previews for HTTP(S) images should use `/api/media/proxy?url=...` so the server can send platform-friendly media headers. This improves preview reliability but does not recover truly expired signed URLs.
- The content-pool `cache_media` batch action can retry downloading existing remote media into `public/media/crawl`, but it cannot recover assets whose platform URL has already expired or no longer authorizes downloads.
- Video local caching only works when the crawler extracted a direct downloadable video URL. HLS/m3u8 or platform-protected playback URLs may remain remote-only unless a different extraction path is added.
- Some Douyin/ByteDance video CDN URLs, including `zjcdn.com/.../video/tos/...` URLs returned for Douyin note imports, can return `200 video/mp4` but abort at 0 bytes when requested with the CDN host as Referer. Use `buildMediaRequestHeaders(...)` from `src/lib/media-request.ts`, which supplies Douyin Referer/Origin and identity transfer encoding for likely Douyin/ByteDance video media; do not hand-roll media download headers in new call sites.
- Video posts with local `videoFrames` should not rely on old remote signed cover URLs for preview; use the extracted frames as the image preview.
- Video-like imported/crawled sources without extracted `videoFrames` may still enter automatic/default production when they have a direct source video reference, because the final draft can carry `GeneratedPost.videoUrls`. Do not use platform cover/origin/dynamic-cover images as a substitute visual source for image tasks; operators who need generated images still need extracted frames, downloaded/source images on image-only posts, or explicit manual review work.
- Historical video/mixed records may still physically contain more than 5 `videoFrames` until they are read, refreshed, updated, or backfilled. The app should cap stale records through `src/lib/video-frame-policy.ts` in preview, tagging, production-task creation, and content-pool writes instead of mutating runtime data during Trellis-only work.
- Video preview requires HTTP byte-range support. The local media API supports Range requests for mp4/mov files; if video previews break again, verify `/media/crawl/.../video-1.mp4` returns 206 for `Range: bytes=0-1023`.
- Weibo video posts can legitimately have `images=[]` while still having cached `videoFrames`; preview and production image selection should treat those keyframes as usable visual references.
- TikHub Weibo App search payloads can contain UI layout arrays such as `{name, layout}` near the real post data. Do not use generic likely-array selection for Weibo; extract `mblog`, `data.mblog`, or direct mblog-shaped records.
- Weibo App image extraction should not use broad raw-record URL fallback. Use content fields such as `pics` and `pic_infos`; otherwise avatars, vote/ad icons, and other decoration URLs can pollute `images`.
- Weibo CDN URLs may serve HEIC bytes even when the local cache path ends in `.jpg`. Media caching should convert cached HEIC bytes to JPEG in place before returning a local preview URL, and visual tagging/local serving must sniff local file bytes instead of trusting the extension MIME.
- Douyin image/text posts can return many URL variants per carousel asset, including `video.cover`, HEIC, JPEG/WebP, and watermarked download URLs. Treat `raw.images` / carousel image records as the primary source and choose one supported JPEG/WebP per asset; otherwise cover/variant URLs can fill the image limit and hide later carousel images.
- TikHub Douyin `content_type=2` image requests can still return video-cover cards. The app should skip records without true Douyin carousel/image fields and strip direct video media from kept image records; if image-only Douyin crawls produce `videoFrames`, recheck this guard first.
- `node:sqlite` currently emits a Node experimental warning; this is expected in Node 24 and did not fail baseline verification.

## Integration Pitfalls

- TikHub, OpenAI-compatible endpoints, and Feishu CLI are external integrations; default baseline must not call them.
- Xiaohongshu TikHub detail requests use App V2 `/api/v1/xiaohongshu/app_v2/get_image_note_detail` and `/api/v1/xiaohongshu/app_v2/get_video_note_detail` for both exact source-link import and keyword-result enrichment. Do not restore removed `web/get_note_info_v4`, `web/extract_share_info`, or `web_v3/fetch_note_detail` fallbacks. TikHub can return HTTP 200 with a failing business envelope such as `data.ok=false` and `data.status=461`; validate that envelope in `tikhubRequest(...)` before normalization. Endpoint migrations must update all call sites and add both positive replacement fixtures and negative old-endpoint assertions.
- Batch source-link import uses TikHub detail/share endpoints for TikHub-backed platforms and local HTML importers for supported source-link-only platforms such as Xiaopeng BBS and Dongchedi. Default Trellis coverage validates endpoint construction/importer wiring only; live URL imports still depend on provider response shape, platform short-link resolution, anti-bot behavior, and the configured OpenAI-compatible safety/tagging path.
- Simple-run link mode also resolves source links through TikHub, then continues into GPT tagging/production, image generation, and Feishu publish. Default browser verification should mock `POST /api/simple/runs`; run a real link-mode simple task only when the user explicitly wants a live end-to-end production test.
- Simple-run Feishu mode calls Feishu CLI to read Base records and download attachments, then continues into GPT tagging/production, image generation, and Feishu publish. Default verification should use `.trellis/verification/feishu_content_import_check.mjs`; run a real Feishu task-number import only when the user intentionally wants a live end-to-end production test.
- Douyin `/note/{id}` source links are not safe to treat as direct video ids. TikHub `/api/v1/douyin/web/fetch_one_video_v3?aweme_id=...` returned `404 Not Found` for `https://www.douyin.com/note/7630774012096674665`; use `/api/v1/douyin/web/fetch_one_video_by_share_url` for `/note` links and unwrap `aweme_detail`/`awemeDetail` in the response.
- Content safety filtering is a post-crawl brand-safety quality gate, not a TikHub request parameter or keyword relevance filter. Keep rules and prompts in `src/lib/source-safety.ts`, and do not move them into `src/lib/tikhub.ts` or the existing no-post-crawl keyword/type filter checks.
- A live PostgreSQL server is also excluded from default baseline. Baseline only performs a static schema check unless the project later adds an isolated PG test service.
- AI source visual tagging depends on the OpenAI-compatible relay accepting image inputs. If image inputs are rejected, record visual tagging as failed/skipped and keep content text tags/ingest moving; do not invent visual tags as a silent fallback.
- `GET /api/config` exposes configuration status and model/base URL names, but not API keys.
- Feishu CLI arguments can still be template-based through `FEISHU_CLI_BITABLE_ARGS`; the default command is now `base +record-batch-create --as bot --base-token {appToken} --table-id {tableId} --json @{recordPayload}`.
- Feishu Base publishing requires both target identifiers: `FEISHU_BITABLE_APP_TOKEN` (Base token) and `FEISHU_BITABLE_TABLE_ID` (table ID). App id/app secret alone only configure the CLI application identity.
- The current target Base fields are `动态标题`, `动态正文`, and attachment field `动态素材`.
- Feishu publish success notifications are optional and require at least one recipient env value: `FEISHU_NOTIFY_CHAT_ID` or `FEISHU_NOTIFY_USER_ID`. If both are configured, `src/lib/feishu-cli.ts` sends to the chat id. The bot must have message-send scope and access to the target chat/user.
- Feishu notification failure should be surfaced as a publish warning, not treated as a failed Base write after the record and attachments have already been created.
- Feishu IM `--idempotency-key` values should stay short. A long generated key caused `HTTP 400: field validation failed`; use the compact `fp-...` key format in `src/lib/feishu-cli.ts`.
- GPT-Image-2 via Packy/OpenAI-compatible Images API does not support `n > 1`; the app must loop one-image requests instead of sending a multi-image `n`.
- `openai_json` uses ordinary JSON `POST /images/generations` and multipart `POST /images/edits` without `stream`, `response_format`, or `input_fidelity`; `openai_sse` is the separate relay-streaming profile. ToAPIs uses `toapis_async` with asynchronous `/images/generations` and URL-only `reference_images`.
- Under the OpenAI dialect, local `/media/...` or `/generated/...` references are normalized to temporary JPG files before multipart upload. Under ToAPIs, local-only files first use `/uploads/images`; public TOS/HTTP URLs pass directly.
- Packy `gpt-image-2` does not support Responses API image generation. Use `OPENAI_IMAGE_ENDPOINT=images` for this provider; stale/unknown image endpoint values normalize to `images`.
- The optional GPT-Image-2 backup route requires both `OPENAI_IMAGE_BACKUP_BASE_URL` and `OPENAI_IMAGE_BACKUP_API_KEY`. Route failover is for provider/transport failures only; do not treat content-safety or invalid-image errors as a signal to switch providers.
- Primary and backup routes resolve `{profile, model}` independently. A `404` usually means the selected profile or `/v1` base prefix is wrong; do not respond by adding catch-all payload fields or hostname guesses.
- Never auto-run `/api/config/image-provider-check`: it intentionally performs two paid generations. The button requires saved config and administrator confirmation; baseline checks verify wiring only.
- ComfyUI Klein is a local single-workflow image path, not a high-throughput remote provider. `WORKER_LOCAL_IMAGE_CONCURRENCY` is intentionally hard-capped at `1`; increasing batch/user concurrency will queue Klein image slots instead of running them in parallel.
- ComfyUI Klein is opt-in. A configured workflow path or inline API JSON alone must not route tasks to local ComfyUI; `COMFYUI_KLEIN_ENABLED=true` is also required. With the default `false`, car-exterior and people-with-car selected source-image tasks should stay on the GPT-Image-2/OpenAI Images API path.
- The configured Klein workflow file may be a ComfyUI UI workflow JSON rather than an API prompt JSON. Conversion must preserve `extra.ue_links` from Anything Everywhere nodes, otherwise hidden model/VAE wiring can be lost and ComfyUI prompt validation can fail.
- `COMFYUI_KLEIN_WORKFLOW_PATH` is read for each Klein task, so changing the workflow JSON on disk can affect subsequent tasks without code changes. Changing env overrides such as KSampler steps/cfg/seed requires restarting the app so `src/lib/config.ts` reloads the values.
- `image_generation_queue` records local Klein job state for observability, but it does not resume a ComfyUI prompt after the Node process is restarted. Avoid restarting the local server while a real Klein image is running unless the user accepts that interruption.
- Default baseline verification must not submit a real ComfyUI prompt. Use `.trellis/verification/comfyui_klein_check.mjs` for static wiring; run a live Klein image only as an explicit manual test.
- Image provider requests must not be submitted with an empty prompt. No-media source items can produce `imageTasks=[]`; if the text model also returns an empty `imagePrompt`, the app must use the simple-mode fallback prompt or skip locally instead of submitting the provider request.
- Viral imitation image tasks are not ordinary best-effort source-image tasks. Each selected task keeps `referencePolicy="strict_dual_reference"` and exactly two ordered references: selected vehicle material first, viral style image second. OpenAI sends two files through `/images/edits`; ToAPIs sends two public/uploaded URLs through `reference_images`. Do not allow failures to fall back to the vehicle image as successful generation; save `needs_review` diagnostics and skip automatic publish.
- ToAPIs task submission and task-status querying are separate phases. After a task id is accepted, transient query errors retry that id; failover must not resubmit the paid generation.
- ToAPIs documentation lists `queued` and `in_progress`, but the deployed route can return `pending`; treat `pending` as non-terminal and continue the same task poll.
- ToAPIs result URLs expire after 24 hours. Download and persist them through the runtime-media boundary immediately after `completed`; never store the temporary provider URL as the post's durable media.
- Default baseline verification must not submit real GPT-Image-2 generation tasks. Use static request-shape checks, local config/status checks, and the project baseline unless the user explicitly asks for a paid/live generation test.

## TOS Runtime Media Pitfalls

- Verified TOS uploads delete local staging files, but generated posts may still hold older `/media/` or `/generated/` snapshots. File-only consumers must recover only the exact same logical TOS object and persist the canonical URL before external writes; never repair by basename or let one missing post abort an otherwise valid Feishu batch.
- `@volcengine/tos-sdk@2.9.1` currently pulls Axios versions with published high-severity advisories and no fixed SDK release. Keep TOS endpoint/public-base values admin-controlled, force SDK HTTPS with certificate verification, disable proxy use and SDK retries, redact AK/SK from errors, and re-evaluate the dependency when Volcengine publishes an updated SDK.
- After adding a dependency on Windows, do not assume a local build proves `package-lock.json` is Linux-complete. npm can omit Linux optional peer entries such as top-level `@emnapi/core` and `@emnapi/runtime`; regenerate the lock with the deployment Node image and run Linux `npm ci --dry-run` before pushing a dependency-changing deployment.
- This Windows development environment has been observed with `NODE_TLS_REJECT_UNAUTHORIZED=0`. That setting weakens TLS globally and must not be copied to the VPS; verify it is unset in the FluxPost container before enabling TOS.
- A successful PUT alone is insufficient. Accept a media URL only after HEAD returns the expected byte length and a non-empty ETag; otherwise the only recoverable copy must remain in `data/tos-pending`.
- Do not solve an anonymous GET failure by changing the whole Bucket ACL. The production contract requires object-level `public-read`; keep `TOS_ENABLED=false` until the admin probe passes.
- Reconciliation uploads retained bytes but intentionally does not replay or mark the original crawl/generation/import task successful. An operator must review and retry the business workflow after storage recovery.

## Verification Pitfalls

- Do not invent `npm test`; no `test` script exists in `package.json`.
- Run baseline from the project root.
- If the smoke port is occupied, set `TRELLIS_SMOKE_PORT` to a free local port and rerun the same baseline script.
- Do not replace a failing check with a softer fallback. Fix the real failing command, data parse issue, build issue, or smoke assertion.
- In the current workspace sandbox, `npm run build` can pass directly while `Next build` inside `.trellis/verification/check.ps1` fails with `spawn EPERM`; rerun the same baseline with approved elevated permissions before changing checks.
- Playwright browser subprocess startup can fail in the sandbox with `WinError 5`; use approved elevated permissions for local browser UI self-checks.
- If Chrome/Edge CDP checks need a custom `--user-data-dir`, do not leave the browser profile directory under the project tree before running baseline. ESLint can scan extension JavaScript from those profiles. Screenshots under `test-artifacts/` are fine, but temporary browser profile folders should be outside the project or removed before `npm run lint` / `.trellis/verification/check.ps1`.
